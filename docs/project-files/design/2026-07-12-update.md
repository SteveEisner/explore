# Design Update — 2026-07-12

This is shifting from a simple explain-my-docs mode into an artifact studio.

It doesn’t just describe — it helps shape the artifact, then builds it in place. The artifact is a shareable document that contains the app, and the .oui files are the apps.

There’s also a source layer: a markdown file that captures intent and origin. Shared output is a full HTML artifact, with a view-source trail of where it came from.

So it’s one artifact, but with more context than a typical finished output. Because it carries its origins, you can ask it better questions.

One-hour plan: tighten the chat-and-tools editing loop so relevant docs are easier to find, screenshots map cleanly to UI parts, and small edits apply quickly.

Then add a view-source mode. The app opens as the interactive .oui experience, with a clear switch to a markdown source view and back.

## Upfront decisions and assumptions

### Early decisions
You chose to go beyond a quick two-hour demo and build something more complete and reusable. You also decided to set up a minimal multi-agent approach rather than relying on a heavier existing framework. You planned the work in phases: establish taste and structure, do a short blitz, then clean up. You treated the backend as a stable set of capabilities protected by a test boundary, while the frontend stayed flexible and disposable as you learned. You prioritized what to build into P1 and P2 buckets, and even though the frontend could be rough at times, you reserved time for polish so the demo would feel solid and something you’d be proud to show.

You also chose not to reinvent the wheel where it didn’t help: use the filesystem as the wiki, rely on an existing intelligence backend, and use a voice model as the front-end intelligence. The intentional exception was the authoring frame itself — you built a surface that overlaps with Claude’s own co-working and code-assistant tooling, accepting some redundancy even though you’re already calling into Claude, so you can control the environment and extend it inside this web app.

### Things you already knew (and used)
This is an honest acknowledgement that you built on top of a lot: a markdown renderer, an OpenAI voice agent, artifact generation and code review capabilities, a small agent setup, and a Vite scaffold running on Node and TypeScript you already trusted. You didn’t have to research those from scratch, so the real effort was wiring them together and shaping the app.

But there was substantial work after that foundation. You shipped inline artifact support as the preferred way to build interactive views inside documents, refined the voice-driven editing experience and its "hands on the computer" persona, and shaped the UI flow around previews, expanding artifacts, and a tighter chat/toolbar experience. You also built and exercised the underlying system wiring — state store behaviors, tool-driven edits, and end-to-end flows — and captured the decisions so they’re repeatable.
