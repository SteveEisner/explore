PR 502764: Shared Billing Collector Apply Flow

A walkthrough of the collector-model refactor in apply-time billing.

Context

From Stack To Code

Summary

This PR makes sure that when a customer’s subscription changes, the system does not lose track of future billing work. Some charges happen immediately, but usage-based charges may need to happen later. This change makes the order-apply step decide, for each existing future billing instruction, whether to keep it, replace it, move it forward, or remove it.

Components

This change is in Order Workflow, the part of Cash billing that applies an accepted order. In plain terms, this is where the system turns a planned subscription change into saved bills, saved subscription updates, and saved instructions for later billing.

Change Context
Distinct Changes

One apply-time builder now assembles PREPAID future rebills and POSTPAID collector outcomes before the replacement change plan is saved.

POSTPAID collectors are resolved by item id, so each saved collector is advanced, rewritten, preserved, consumed, or dropped.

Not Changed

Preview bill calculation, product pricing, storage schema, and storage APIs stay outside this slice.

Apply-time flow

Apply starts with the bills and timelines preview already produced. It saves the bill rows that belong to this order, then saves the future billing instructions that should remain after the order changes the subscription.

Apply saves immediate bills first, then replaces the future-action change plan. The replacement plan has to include PREPAID future rebills and POSTPAID usage collectors because storage reads one latest future-action set after apply.

Read the flow through `apply_bills`, `_create_future_billing_collector_actions`, `_create_postpaid_future_actions`, and `_preserve_postpaid_actions_for_subscription`. The control flow matters because old collector state is needed before cleanup, and cleanup is needed after replacements are saved.

The flow is correct only if every collector observed in the pre-cancel change plan is either consumed, advanced, rewritten, preserved, ignored as PREPAID, or dropped because it has no valid future window. That outcome has to happen before storage latest state is allowed to forget the old revision.

Callouts

These are the parts of the change where the code has to preserve a non-obvious fact across several steps. Each callout names the fact, the source of risk, and the evidence a reviewer should look for.

One POSTPAID item has one active collector sequence

Highest risk

A saved POSTPAID collector can be reached from more than one part of apply. Bill items show windows that were just billed. Current timeline items show what remains active. Preservation sees saved collectors that neither path handled.

The risk is duplicate or stale future work: two branches could create a successor for the same item, or one branch could preserve a collector another branch already consumed. The code keys old collectors by subscription item so each branch is talking about the same collector sequence.

The invariant is stronger than “no duplicate actions in the returned list.” It is “no duplicate active future work for one POSTPAID subscription item after storage reconstructs the latest change plan.”

Before

PREPAID future actions are built from NEXT_REBILL bills. POSTPAID saved collectors are not resolved as item-level outcomes here.

After

old_postpaid_actions_by_item_id for immediate bill item: consume old collector and add one successor for current POSTPAID item: rewrite, advance, create, or drop collector preserve remaining collectors only if still open

Old collector state is read before cleanup removes it from latest state

Ordering

Replacement builders need the old actions to know which usage window is open, consumed, or due. Cleanup also has to remove old latest revisions.

The risk is ordering-dependent: if cleanup runs before the code snapshots the old change plan, the old collector window may no longer be available to build the replacement. If cleanup does not run after replacements are saved, stale old revisions can remain active.

PREPAID and POSTPAID still use different evidence

Boundary

The helper creates one saved payload, but it does not collapse the two billing models. PREPAID future actions come from NEXT_REBILL bill items. POSTPAID collectors come from saved usage windows and current POSTPAID timeline state.

The risk is a false simplification: because both action types are saved together, it is tempting to treat them as if they are derived the same way. The code should share the final payload construction, not the source-of-truth rules.

PREPAID path

NEXT_REBILL bill item describes the next coverage block. Saved REBILL action supplies the previous block for continuity.

POSTPAID path

Saved collector describes the usage window. Immediate bills consume windows; current timelines decide whether a future collector still exists.

Evidence

These tests are useful when they prove a saved-state outcome, not when they merely exercise a branch. Read each case as: given this collector state before apply, storage contains this collector state after apply.

The compact review question

For every saved or current POSTPAID collector state the code can observe, the outcome table should produce exactly one correct storage result.

That question also tests the size of the change: the PR changes apply-time collector persistence while leaving preview calculation, storage schema, and billing frequencies outside the diff.
