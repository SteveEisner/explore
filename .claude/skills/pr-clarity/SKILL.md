# PR Clarity Guidelines [SteveE]

## Overview

Turn a PR description into a compact review guide. Inspect the code and PR state first; do not merely rewrite commit subjects or force the body into a fixed format.

The target description should help reviewers understand why the PR exists, what changed, how to inspect it, what is intentionally out of scope, what risks remain, and which tests support the claims.
Optimize for comprehension and reviewability: the body should reduce the work needed to reconstruct the diff's purpose, boundary, correctness, and safe review path.
Keep that reviewability work out of the PR's stated purpose. The PR body should narrate the change itself in domain, behavior, correctness, clarity, or validation terms, not describe the PR process or claim that the purpose is to make a PR easier to review.

Use the skill in author mode when creating or updating the PR's review surface. Use it in reviewer mode when the user wants to understand an existing PR: explore the PR and explain the narrative that the PR body should have provided, without updating the PR unless explicitly asked.

Deep review means using the available Clarity skills as review lenses, then reconciling their findings against the PR narrative. Use `code-clarity` for implementation and configuration, `comment-clarity` for comments, docstrings, and written intent, and `test-clarity` for tests, fixtures, and proof quality. Cross-check the code, comments, tests, and PR narrative as one contract: the implementation should do what the PR claims, comments should describe the current local behavior and invariants, tests should prove the important cases, and all four surfaces should agree about correctness, scope, and risk.

## Workflow

1. Identify the PR and branch.
   - Use the PR number/link from the user when provided.
   - Otherwise inspect the current branch and ask only if there are multiple plausible PRs.
   - Read the existing PR title/body with `gh pr view <num> --json number,title,body,headRefName,baseRefName,url`.

2. Inspect the actual change.
   - Run `git status --short --branch` first so branch state is explicit.
   - For a stacked PR, compare against the direct parent branch when known; otherwise compare against the PR base.
   - Read `git diff --stat`, changed filenames, relevant hunks, and the commit log.
   - Inspect live review state when reviewing an existing PR: unresolved GitHub review threads, author replies, and prior reviewer guidance can change what still needs to be surfaced.
   - Inspect outside-the-diff context that changes how reviewers should understand this PR: parent branches, feature flags, current rollout state, config defaults, migrations, generated-code sources, architecture docs, design docs, or upstream contracts.
   - If the PR is stacked, inspect adjacent PRs/branches enough to avoid describing inherited changes as new.

3. Determine the review thesis.
   - State what problem this PR solves in one paragraph.
   - Separate runtime behavior, schema/model/API shape, tests, and documentation.
   - Distinguish launched behavior from scaffolding or future-use primitives.
   - Do not overclaim: if production behavior is gated, defaulted off, or not wired yet, say so.
   - Identify the correctness and validation narrative: the obligation the diff must satisfy, the model or state space that makes the obligation tractable, which evidence supports each important case, and what remains unproven or out of scope.
   - For initial PRs and deep PR reviews, do a fast clarity sweep across code, comments, and tests. Surface P1 clarity issues to the author without blocking PR creation; suggest a focused follow-up when the issue should be addressed quickly.
   - Identify any reviewability recommendations separately from the PR description itself.

4. Write only the sections that help this PR become easier to review.
   - Always satisfy the summary and verification obligations, but choose headings and sections based on what this PR needs.
   - Add design rationale, flow maps, changed-area bullets, minimality notes, deployability/risks, context boundary, and stack context only when they help reviewers inspect the diff.
   - Combine changed-area bullets and review-path guidance when separate sections would repeat the same files or concepts. A combined section should name what changed, why that area matters, and what the reviewer should inspect there.
   - Group changes by concept or subsystem, not by every file.
   - Use Mermaid diagrams, compact flow maps, tables, or other visual aids when they make a complex control flow, state machine, data flow, stack relationship, migration path, or correctness model faster to understand. A super-simple diagram that only restates obvious prose is noise, but a clear picture is often worth more than another dense paragraph when the code has real branching or sequencing complexity.
   - For stacked PRs, mark "This PR" clearly and keep inherited work out of this slice's description.

5. Add targeted GitHub comments when they materially improve review.
   - For newly created PRs, consider adding comments on especially tricky or meaningful hunks that benefit from local review narration.
   - Use comments for diff-local meta-narrative that does not belong in persistent code comments: "this was removed because...", "this now handles...", "this is intentionally unchanged because...", or "review this with the generated output below."
   - Keep comments sparse and high-signal. Do not annotate obvious code or duplicate the PR body.

6. Update the PR.
   - When creating an initial PR, create it as a draft unless the skill invocation explicitly says the PR should be ready for review.
   - Prefer a non-interactive update path. For GitHub API:
     `gh api repos/<owner>/<repo>/pulls/<num> -X PATCH -f title='<title>' -F body=@/tmp/pr-body.md`
   - Verify with `gh pr view <num> --json title,body,url`.
   - Do not create commits or change working-tree files for a description-only update.
   - If you have recommendations that require code, commit, branch, or test changes, report them separately instead of applying them.

## Acting As Author

Use this mode when creating an initial PR, updating a PR description, adding targeted GitHub comments, or making the PR easier to review on behalf of the author.

- Apply the workflow above and update the PR only after inspecting the actual diff and context boundary.
- Create initial PRs as drafts unless the invocation explicitly says ready for review.
- Keep the PR body factual and reviewer-facing. Put sparse diff-local meta-narrative in GitHub comments when it helps reviewers understand especially tricky or meaningful hunks.
- For initial PR creation, check for P1 code, comment, and test clarity issues. Do not refuse to create the PR because of them, but report them to the author and suggest the fastest useful follow-up.
- Keep reviewability recommendations that require code, commit, branch, or test changes separate from the PR body unless the user asks to include them.

## Acting As Reviewer

Use this mode when reviewing or exploring a PR you do not own, when the user asks what a PR does, or when the user wants the missing reviewer narrative without editing the PR.

- Do not update the PR, create comments, change code, or critique the author's style unless explicitly asked.
- Explore the PR enough to reconstruct the review guide it should have given: purpose, context boundary, causal narrative, inspection path, correctness notes, risks, validation, and intentionally unaffected behavior.
- Explain what you would have written in the PR description, but present it as a reviewer aid rather than as a required rewrite.
- Favor a narrative the reviewer can follow without loading the code in an editor. Include file and hunk entry points only where they help the reader inspect the important behavior.
- For deep PR reviews, also surface P1 code, comment, and test clarity issues that materially affect review confidence. Keep this separate from the reconstructed PR narrative.
- Account for unresolved GitHub comments the author may not have seen yet. Treat unresolved threads as part of the current review context, but do not repeat guidance that the author has already acknowledged, answered, or been separately told about.
- Separate facts learned from the diff from inferences about intent, rollout state, or safety. Name any context that could not be verified.
- If the PR body is stale, incomplete, or misleading, say what narrative is missing; do not turn the response into a stylistic critique.

## Description Standards

Write for a reviewer who is smart but not yet loaded into the branch.

- A good code change lets reviewers judge four things: whether the stack solves the right problem, whether the chosen approach is the right solution, whether this PR implements its slice correctly, and what durable lesson or follow-up the author should carry forward. Apply this as a principle, not a rigid checklist: a narrow PR may only need to answer its local slice, and "leave the codebase better" often means preserving surrounding code unchanged instead of tidying unrelated paths.
- Lead with the user-facing or system-facing outcome, not implementation trivia.
- Avoid PR meta-narrative in the PR body. Do not make the stated purpose about the review process, the PR artifact, or how the PR is easier to inspect. It is fine for a change to improve clarity or reviewability, but express that as an outcome of the code or system: clearer ownership, simpler state transitions, narrower invariants, more direct validation, or more provable correctness.
- Avoid commit archaeology in the initial PR narrative. Reviewers usually need the current purpose, boundary, correctness notes, and validation more than the sequence of how the branch got there. In later review phases, it can be useful to mention archaeology when it helps a reviewer who already saw an earlier version: for example, a change of mind based on review comments, a specific follow-up that changes the validation story, or a narrowed risk that explains why rereview should focus somewhere else.
- Introduce acronyms, domain concepts, state transitions, and upstream spec assumptions before relying on them.
- Make the narrative reviewable without requiring jumps to specs, lower-stack PRs, helper definitions, or generated code for basic orientation.
- Order the narrative from cause to consequence: why this exists, what contract or state changes, how the implementation follows, what is intentionally unaffected, and how the result was verified.
- Remove redundancy between sections. Each section should add a distinct piece of reviewer value: summary orients, changed-area bullets name the concrete delta, review guide gives the inspection path, context bounds the slice, correctness notes explain invariants/risks, and verification states current validation. Keep wording concise and understandable instead of repeating the same claim under multiple headings.
- Scale the body with narrative complexity, not with a fixed template. A PR description should say only what a reviewer cannot quickly learn from the diff itself. Even a short body should still make the change's necessity and correctness judgeable when they are not obvious from the patch: why the old behavior was insufficient, what invariant the new behavior relies on, and what evidence proves the important case. If the code comments explain the local invariant, do not repeat that invariant in full; point reviewers to the relevant block or summarize the review decision they need to make. For a small self-contained change, a few connected sentences plus validation can be enough. Add headings, diagrams, correctness notes, or context only when the reviewer needs extra domain model, state flow, risk boundary, or stack context to judge the change.
- When changed-area and review-guide sections overlap, prefer one section such as `Change And Review Path`. Each bullet or paragraph should pair the concrete delta with the inspection path: the changed component, the reason it matters, and the claim the reviewer should verify there.
- State the PR's correctness obligation: what reviewers should be able to verify in the diff, what invariant is preserved, or what behavior becomes safe. Connect important claims to evidence in readable prose rather than turning the body into a list of citations.
- Give reviewers an inspection path: where to start, what that entry point establishes, what to read next, and which files are mechanical, generated, or supporting.
- When naming a method, function, class, test, or other local symbol, include enough file context for a reviewer to find it when the surrounding text has not already done so. GitHub review does not make jump-to-declaration reliable for every reviewer, but repeated file links and full repo paths are often noisy when all changed files sit in the same few folders. The summary section can usually orient without file references because the reviewer is not yet taking action; file context matters more in review guides, correctness notes, and targeted bullets that tell reviewers where to inspect. Prefer a short disambiguating reference such as `method_name` in `order_workflow_billing_processor.py`; after nearby text has established the file, bare symbol names are fine. Use a longer relative path only when the filename is ambiguous or the directory matters.
- Keep bullets flat and skimmable.
- Document what is intentionally not included in this PR.
- Include deployability details when relevant: defaults, gates, additive-only fields, no-op behavior, migrations, alerts, or rollback shape. Do not say a change is safe without naming the guard or evidence that makes it safe.
- Keep verification claims accurate and reviewer-useful. Describe what the reviewer should know is true at the time they read the PR, summarizing successful validation at the level reviewers need, such as test suites, lint/typecheck families, migrations, generated-code sync checks, or targeted cases that prove a specific invariant. Avoid chronological "before this / after that" validation sections unless timing materially changes the risk. Avoid listing PR-management commands, mechanical push/update actions, exploratory commands, or every failed attempt used while preparing the branch.
- Keep failed validation out of the PR description unless the PR intentionally documents a known failing state. Surface failed, skipped, flaky, or blocked checks to the person creating the PR, with enough detail to decide whether to fix, rerun, or disclose a remaining risk.
- Call out large generated, mechanical, or boilerplate portions of the diff when reviewers can safely skim them, and point to the source files or commands that produced them when useful.

## Correctness Notes

Use correctness notes to give reviewers a concise, readable reason to trust the change. The section should be simpler than the code it explains; use structure only when it makes the claim easier to inspect.

- State the main correctness obligation in system terms: the invariant, state transition, safety property, compatibility contract, or user-visible behavior the PR must preserve or establish.
- Give reviewers the smallest useful model for that obligation. This might be a state machine, ledger, request lifecycle, data-flow boundary, migration phase, ordering rule, compatibility matrix, or finite set of input/output cases.
- Cover the important cases in that model and the expected outcome for each. A compact table or grouped bullets can help when several cases share one mechanism, but plain prose is fine when it stays clear.
- Connect evidence to the cases it supports. Tests, assertions, static checks, integration suites, comments, or generated-code checks are most useful when the reader can see which state, transition, or invariant each one backs up.
- Name residual risk or intentional boundaries in the same terms as the model. If inherited behavior, integration coverage, manual reasoning, or an intentionally unproven case matters to review, say so directly.
- Keep the explanation at the reviewer-decision level. Include enough file or symbol context to inspect the claim, but leave low-level assertion details to the tests unless a particular assertion is the key evidence.

## Context Boundary

Describe outside-the-PR context only when it changes how reviewers should understand this PR. The goal is the slice boundary, not a full project history.

- Identify parent PRs/branches for stacked work and compare against the direct parent when summarizing the diff.
- Identify important code, configuration, flags, current rollout state, migrations, or upstream contracts that affect this PR but are not included in it.
- Include known directly related specs, design docs, RFCs, tickets, architecture docs, or other docs when they help reviewers understand the context boundary, intended behavior, or validation standard. Folder-level architecture docs such as `ARCHITECTURE.md` are especially useful when they apply to the changed code or when the PR updates them as part of the contract. Keep this to the docs that materially inform review; do not turn the PR body into a project bibliography.
- Include inherited dependencies only when they explain this PR's boundary, deployability, correctness, or validation.
- Avoid claiming files changed in the GitHub PR view are part of this slice if they come from lower branches.
- Replace inherited commit-subject lists with this PR's responsibility.
- Skip context that does not help reviewers inspect or validate this PR.

## Minimality Review

Before updating the body, look for review-hostile churn. Mention the highest-value answers when they help reviewers know where to focus:

- Could this have been a narrower branch?
- Are equivalent refactors mixed with behavior changes?
- Did the implementation preserve existing control-flow shape where possible?
- Are new helpers justified by reuse or clarity?
- Are tests proving the invariant instead of mirroring implementation details?
- Does the PR body have to work too hard because the diff itself is tangled?

## Reviewability Recommendations

The skill may recommend follow-up changes, but updating a PR description must not require or perform code changes.

Check for:

- Stale description content that no longer matches the diff.
- Unrelated changes or mixed concerns.
- Mechanical, generated, or boilerplate-only files mixed with core behavior changes.
- Missing reviewer entry points.
- Missing or unclear test evidence.
- Safety, deployability, or "no behavior change" claims without supporting evidence.
- Unresolved review threads that materially affect the PR narrative, validation story, or author follow-up and have not already been acknowledged or handled.
- A useful lesson, documentation gap, or follow-up pattern that should be surfaced to the author even when it does not belong in the PR body.
- A PR that is too large or tangled to explain honestly.

If the PR is too large to make reviewable with notes, say so and recommend a split rather than polishing around the problem.

## Title Guidance

Use a title that names the reviewable behavior or contract delta, not the mechanics. Stack numbers, product names, and subsystem prefixes are useful only when they help route or frame review.

Good:

- `[5] Usage Based Billing: scheduled POSTPAID rebill primitives`
- `Support cancelled scheduled billing actions`
- `Usage Based Billing: rating/pricing implementation`

Avoid:

- `Clean up more`
- `Fix comments`
- `Update files`
- A title copied from a lower stacked PR
